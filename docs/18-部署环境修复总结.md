# 部署环境修复总结

日期: 2026-05-22

## 1. Docker 部署中文乱码

**问题**：Docker 部署后，员工姓名等中文数据在 API 返回中变成乱码（如 `系统管理员` → `ç³»ç»Ÿç®¡ç`）。

**排查过程**：
1. MySQL 容器中直接查询数据 — 中文正常，确认存储没问题
2. 查看 MySQL 连接编码 — `character_set_client = latin1`，`character_set_results = latin1`
3. JDBC URL 中 `characterEncoding=UTF-8` — Connector/J 8.3.0 中已废弃，不生效
4. `connectionCollation=utf8mb4_unicode_ci` — 理论上应生效但实际不工作
5. `connection-init-sql: SET NAMES utf8mb4` — HikariCP 未正确执行
6. `--init-connect="SET NAMES utf8mb4"` — 对 root 用户无效（root 有 SUPER 权限跳过 init_connect）

**根因**：MySQL 8.0 Docker 镜像默认客户端连接编码为 `latin1`，而开发环境的 MySQL 默认是 `utf8mb4`。

**最终方案**：`docker-compose.yml` 中 MySQL 容器启动参数加 `--skip-character-set-client-handshake`

```yaml
command: ["--character-set-server=utf8mb4", "--collation-server=utf8mb4_unicode_ci", "--skip-character-set-client-handshake"]
```

该参数强制 MySQL 忽略客户端发送的编码请求，统一使用服务器端编码（utf8mb4），对所有用户（包括 root）生效。

## 2. 文件上传 413 错误

**问题**：几十 KB 的图片上传成功，几 MB 的上传报 413。

**根因**：Nginx 默认 `client_max_body_size` 仅 1MB。

**修复**：`inventory-admin/nginx.conf` 加 `client_max_body_size 10m;`，与后端 `spring.servlet.multipart.max-request-size: 10MB` 对齐。

## 3. 数据库密码明文硬编码

**问题**：`application.yml` 中数据库密码 `123456` 和 JWT 密钥明文写死。

**修复**：改为环境变量占位符，开发环境用默认值，生产环境通过环境变量注入：

```yaml
spring:
  datasource:
    username: ${DB_USERNAME:root}
    password: ${DB_PASSWORD:123456}

sa-token:
  jwt-secret-key: ${JWT_SECRET:inventory-system-jwt-secret-key-2024}
```

**生产部署时设置环境变量**：
```bash
# 方式一：启动命令前
DB_PASSWORD=真实密码 JWT_SECRET=随机32位字符串 java -jar app.jar

# 方式二：Docker Compose environment
environment:
  DB_PASSWORD: 真实密码
  JWT_SECRET: 随机32位字符串
```

## 4. 前端 TypeScript 类型错误

**问题**：Docker 构建 `npm run build` 失败，`vue-tsc -b` 报 45 个类型错误。

**修复**：
- tsconfig 放宽 `noUnusedLocals/noUnusedParameters`（消除 25 个无关错误）
- ElMessageBox API 从三参数改为两参数（Element Plus 2.13 兼容）
- `type: 'danger'` → `type: 'error'`（Element Plus 2.13 合法值）
- 详情页 data ref 改为 `any` 类型（approverName/approveTime 等新增字段）
- Form 中回调参数加显式类型 `(val: any)`
