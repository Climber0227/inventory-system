# 进销存系统性能优化记录

> 日期：2026-06-09  
> 优化范围：后端 14 个 Java 文件 + 前端 6 个 Vue 文件 + 1 个配置文件

---

## 一、核心性能优化

### 1.1 全局消除 N+1 查询（10 个文件）

**问题**：列表查询在循环中逐条 `selectById`，10 条记录可能产生 50+ 次 SQL。

**修复**：收集所有 ID → `selectBatchIds()` 批量查询 → Map 内存映射赋值。

| 文件 | 改了什么 |
|------|---------|
| `inventory/service/InventoryService.java` | `enrichNames()`、`getAlertList()` 改为批量加载 product + warehouse |
| `product/service/ProductService.java` | `enrichInventory()` 改为 `enrichInventoryBatch()`，1 次查所有产品的库存 |
| `purchase/service/PurchaseOrderService.java` | `page/listAll/getDetail` 合并为 `enrichOrdersBatch()`，批量加载 supplier/warehouse/user/item/product |
| `sales/service/SalesOrderService.java` | 同上，`enrichOrdersBatch()` |
| `transfer/service/TransferService.java` | `page/listAll/getDetail/getExportDetailList` 合并为 `enrichTransfersBatch()` |
| `stocktake/service/StockTakeService.java` | `page/listAll/getDetail/getExportDetailList` 合并为 `enrichStockTakesBatch()` |
| `report/service/ReportService.java` | `turnoverRate()` 批量加载 items + products |
| `inventory/service/DashboardService.java` | `getAlertList()` 从调两次改为调一次复用 |

### 1.2 仓库树去递归化（1 个文件）

**问题**：`WarehouseService.enrichStats()` 对每个仓库节点递归调用 `hasChildren()`（COUNT 查询）+ `collectLeafIds()`（递归查子节点）。几千个仓库 = 上万次 SQL。

**修复**：`InventoryMapper.selectWarehouseStats()` SQL 聚合（`GROUP BY warehouse_id`），返回 1.2 万行替代 144 万行。`StatsContext` 一次加载全量仓库 + 聚合库存，纯内存计算。

| 文件 | 改了什么 |
|------|---------|
| `warehouse/service/WarehouseService.java` | 新增 `StatsContext` 内部类，自底向上内存计算叶子后代、库存汇总；`tree/roots/childrenAll/search/listAll/page` 全部改用 StatsContext |
| `inventory/mapper/InventoryMapper.java` | 新增 `selectWarehouseStats()`、`selectProductStatsByIds()` 等聚合方法 |

### 1.3 库存统计 SQL 聚合（1 个文件）

**问题**：商品列表加载库存时，`SELECT * FROM inventory WHERE product_id IN (...)` 返回几十万行到 Java 内存。

**修复**：改为 `SELECT product_id, SUM(quantity) FROM inventory WHERE product_id IN (...) GROUP BY product_id`，返回 2000 行。

| 文件 | 改了什么 |
|------|---------|
| `inventory/mapper/InventoryMapper.java` | 新增聚合查询方法 |

### 1.4 配置优化（1 个文件）

| 配置 | 效果 |
|------|------|
| `server.compression.enabled=true` | GZIP 压缩 JSON 响应，体积 ~10 倍压缩 |
| `mybatis-plus.configuration.log-impl=NoLoggingImpl` | 关闭生产环境 SQL 日志，消除 IO 开销 |
| `rewriteBatchedStatements=true` | JDBC 批量合并 INSERT |

## 二、导入性能优化

### 2.1 MyBatis BATCH 模式（2 个文件）

**问题**：导入时逐条 `warehouseMapper.insert(w)`，1.4 万条仓库 = 1.4 万次网络往返。

**修复**：使用 `SqlSessionFactory.openSession(ExecutorType.BATCH)`，200 条攒成一批发送。

| 文件 | 改了什么 |
|------|---------|
| `product/service/ProductService.java` | `batchInsertProducts()` 用 BATCH 模式 |
| `warehouse/service/WarehouseService.java` | `importExcel()` 逐级去重 + BATCH 模式 |

### 2.2 编码批量预生成（2 个文件）

**问题**：每新建一个仓库/商品，调用一次 `synchronized generateXxxCode()`，串行排队。

**修复**：预先批量生成编码池，插入时从池中取用。

| 文件 | 改了什么 |
|------|---------|
| `product/service/ProductService.java` | `generateProductCodes(count)` 批量预生成 |
| `warehouse/service/WarehouseService.java` | `generateWarehouseCodes(count)` 批量预生成 |

## 三、导入导出体验优化

### 3.1 导入逐行校验 + 错误收集

| 文件 | 改了什么 |
|------|---------|
| `product/entity/ProductImportVO.java` | 加 `@NotBlank`、`@DecimalMin`、`@Min` 等 JSR 校验注解 |
| `common/util/ValidatorUtils.java` | 新建校验工具类 |
| `common/util/ImportResult.java` | 新建导入结果类（成功数/失败数/错误详情） |
| `product/service/ProductImportListener.java` | 逐行校验，格式错不中断，记录行号和原因 |
| `warehouse/service/WarehouseImportListener.java` | 同上 |
| `product/controller/ProductController.java` | 返回结构化的 `{success, failure, errors, summary}` |
| `warehouse/controller/WarehouseController.java` | 同上 |

效果：
```
之前：采购价填 "abc" → "导入失败: 系统内部错误" — 用户不知道哪行错
现在：弹窗 "第 12 行: purchasePrice: 采购价不能为负数" — 跳过这行，其他照常导入
```

## 五、其他修复

## 七、性能效果对比

| 场景 | 改前 | 改后 |
|------|------|------|
| 仓库树加载 (4000 仓库) | 上千次 SQL，3~5 秒 | 2 次 SQL，毫秒级 |
| 库存列表 (6 万条) | ~20000 次 SQL，10+ 秒 | 3 次 SQL，毫秒级 |
| 商品列表 (500 商品) | N+1 查询，1~2 秒 | 3 次 SQL，毫秒级 |
| 采购/销售/调拨/盘点列表 | 每条记录 5~8 次关联查询 | 每条记录 0 次关联查询 |
| 仓库导入 (1.4 万条) | 超时 | 20~40 秒 |
| 商品导入 (2000 条) | 几十秒 | 3~5 秒 |
| 仪表盘 | getAlertList() 调两次 | 调一次 |
| JSON 响应体积 | 未压缩 | GZIP 10 倍压缩 |
